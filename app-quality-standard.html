🏆 Andy's App Quality Standard v1.0

# 适用于所有App开发，每次启动新项目必须遵守

这不是建议，是硬性要求。不达标不上线。

一、产品定义标准

1.1 立项前必须回答的5个问题

用户是谁？ — 具体到人物画像（年龄、职业、场景），不是"所有人"

痛到什么程度？ — 用户现在在用什么workaround？愿意花多少钱/时间解决？

为什么是我？ — 技术壁垒或时机窗口在哪？竞品为什么没做好？

最小验证是什么？ — 用户用完会说的那句话是什么？（"终于有人做了这个"）

怎么赚钱？ — 第一天就要想清楚变现路径，不是"先做大再说"

1.2 MVP定义原则

  • 一个核心动作：用户打开App只需要做一件事（扫描/输入/拍照/选择）
  • 10秒出结果：从打开App到看到价值，不超过10秒
  • 零学习成本：不需要教程、引导页可跳过、第一次用就会
  • 2周必须上线：超过2周的MVP不是MVP

二、用户体验标准

2.1 第一印象（0-3秒）

  • ☐ 打开App 1秒内看到核心界面（不是splash screen等3秒）
  • ☐ 视觉第一眼就知道这个App做什么
  • ☐ 配色专业、不廉价、有品牌感
  • ☐ 无任何"请允许通知/位置/相机"弹窗打断（需要时再问）

2.2 核心流程（3-30秒）

  • ☐ 核心操作 ≤ 3步完成（不是3个页面，是3次点击/交互）
  • ☐ 每一步都有即时反馈（动画/震动/声音/文字变化）
  • ☐ 加载状态用skeleton/shimmer，不是转圈
  • ☐ 错误提示用人话+给出解决方案，不是error code
  • ☐ 结果展示清晰：好=绿色✅ 坏=红色❌ 注意=黄色⚠️

2.3 情感设计

  • ☐ 完成任务时有微妙的成就感（动画、emoji、congratulations）
  • ☐ 空状态不是白屏，有引导和温暖文案
  • ☐ 出错时语气友善，不blame用户
  • ☐ 关键时刻有surprise & delight（意想不到的小细节）

2.4 性能底线

  • ☐ 冷启动 < 2秒
  • ☐ 核心操作响应 < 200ms
  • ☐ 列表滚动 60fps 无卡顿
  • ☐ App体积 < 50MB（最好 < 30MB）
  • ☐ 离线可用（至少核心功能）

2.5 无障碍

  • ☐ 最小可点击区域 44×44pt
  • ☐ 文字对比度符合WCAG AA（4.5:1）
  • ☐ 支持系统深色/浅色模式
  • ☐ 支持Dynamic Type（系统字号调节）
  • ☐ VoiceOver/TalkBack基本可用

三、视觉设计标准

3.1 设计系统

每个App必须定义：

品牌色（1个主色+1个强调色）
中性色（5级灰度：bg/surface/border/text-secondary/text-primary）
功能色（success绿/warning黄/error红/info蓝）
圆角半径（small: 8px / medium: 12px / large: 16px / full: 9999px）
阴影层级（subtle / medium / strong）
间距系统（4px基数：4/8/12/16/24/32/48/64）
字号系统（caption:12 / body:14-16 / subtitle:18-20 / title:24-32 / hero:40-72）

3.2 视觉质量检查

  • ☐ 没有对不齐的元素（用8px grid）
  • ☐ 没有挤在一起的内容（最小间距16px）
  • ☐ 没有孤立的1px边框或粗糙阴影
  • ☐ 图标风格统一（全用outline或全用filled，不混搭）
  • ☐ 颜色使用克制（单屏不超过3种颜色+中性色）
  • ☐ 对标水准：不低于同类App Store前10名的设计

3.3 动画标准

  • 时长：150-300ms（太快看不见，太慢觉得卡）
  • 缓动：ease-out进入，ease-in退出，spring弹性（Flutter默认curve好用）
  • 场景：页面切换、按钮反馈、列表增删、状态变化必须有动画
  • 底线：宁可没有动画也不要卡顿的动画

四、技术标准

4.1 Flutter项目结构

lib/
  app/          — App入口、路由、主题
  features/     — 按功能模块分（每个feature独立文件夹）
    feature_x/
      view/     — UI页面
      widgets/  — 该feature专属组件
      model/    — 数据模型
      service/  — 业务逻辑/API
  shared/       — 跨feature共享
    widgets/    — 通用组件
    theme/      — 配色/字体/间距
    utils/      — 工具函数
    constants/  — 常量
  l10n/         — 国际化

4.2 代码底线

  • ☐ 零编译warning
  • ☐ 所有公开API有注释
  • ☐ 无硬编码字符串（全走l10n或constants）
  • ☐ 无硬编码颜色/字号（全走theme）
  • ☐ 状态管理统一（Riverpod或Bloc，项目内不混用）
  • ☐ 网络请求有loading/error/empty三态处理
  • ☐ 敏感数据（API key等）不在代码里

4.3 发布前Checklist

  • ☐ App图标设计完成（1024×1024，所有圆角尺寸）
  • ☐ App Store截图 5-8张（6.7" + 6.5" + 5.5"）
  • ☐ App描述写好（关键词前置、features bullet points）
  • ☐ 隐私政策页面上线
  • ☐ 崩溃监控接入（Firebase Crashlytics或Sentry）
  • ☐ 基础数据分析接入（核心事件：打开/核心操作/付费）
  • ☐ 深色/浅色模式都测过
  • ☐ 小屏(iPhone SE)和大屏(iPad)都测过

五、增长标准

5.1 ASO（App Store Optimization）

  • ☐ App名称含核心关键词（30字符内）
  • ☐ 副标题含次要关键词（30字符内）
  • ☐ 描述前3行是核心卖点（大部分用户不展开看）
  • ☐ 关键词字段填满100字符，用竞品名+痛点词+同义词
  • ☐ 第一张截图就是核心功能展示（不是logo）
  • ☐ 截图有文案overlay（大字说功能，小图展示UI）
  • ☐ 有preview video（15-30秒展示核心流程）

5.2 内置增长机制

每个App必须包含至少3种增长机制：

分享功能 — 核心结果一键分享（生成精美卡片图，带App水印+下载链接）

邀请奖励 — Refer a friend得Pro功能/延长试用

Widget — iOS/Android桌面Widget，每天被看到=每天被提醒

Review提示 — 在用户完成核心操作3次后弹出（不是一打开就弹）

通知策略 — 有价值的通知（提醒/结果/洞察），不是骚扰

5.3 病毒传播设计

  • 分享出去的内容本身要有价值（不是广告）
  • 分享卡片要漂亮到接收者想点开
  • 接收者点开能preview价值，不需要先下载
  • 从看到→下载→首次使用 的摩擦力最小化

5.4 留存设计

  • ☐ Day 1有通知/邮件（欢迎+教一个技巧）
  • ☐ Day 3有通知（"你知道这个功能吗？"）
  • ☐ Day 7有通知（"你的周报/总结"）
  • ☐ 有streak/连续使用激励（不一定是gamification，可以是数据积累可视化）

六、变现标准

6.1 定价原则

  • 永远有免费版：让用户先体验价值，再决定付费
  • 免费版必须有用：不是阉割到不能用，是"能用但想要更多"
  • 价格锚定：月付×12 > 年付价格，让年付显得值
  • 首次付费门槛低：周付/月付优先，$3.99-$9.99起步
  • 一次性购买选项：反订阅趋势明显，至少提供lifetime选项

6.2 付费墙设计

  • ☐ 付费墙出现在用户已体验到价值之后（不是注册时）
  • ☐ 清晰展示Free vs Pro区别
  • ☐ 有免费试用（3-7天）
  • ☐ 付费墙本身设计精美（不是系统默认弹窗）
  • ☐ 关闭按钮明显可见（不耍手段）

6.3 变现多元化

除了订阅，至少考虑1-2种：

  • Affiliate推荐（推荐相关产品/服务）
  • B2B版本（企业/团队付费）
  • 数据洞察（匿名聚合数据卖报告）
  • 内容/模板市场（用户贡献+平台分成）

七、合规标准

7.1 每个App必须有

  • ☐ 隐私政策（GDPR + CCPA compliant）
  • ☐ 使用条款
  • ☐ 免责声明（如涉及健康/法律/财务建议）
  • ☐ 数据删除功能（GDPR Article 17要求）
  • ☐ 未成年人保护（如目标用户含<13岁，需COPPA合规）

7.2 数据原则

  • 只收集必要数据
  • 默认不追踪（opt-in不是opt-out）
  • 数据存储加密
  • 清晰的数据使用说明
  • 提供数据导出功能

八、发布后标准

8.1 第一周

  • ☐ 每天查看崩溃报告，24小时内修复critical crash
  • ☐ 回复所有App Store评价（尤其1-2星）
  • ☐ 收集前100个用户的反馈（主动问，不等着）
  • ☐ 追踪核心指标：DAU、核心操作完成率、D1/D7留存、付费转化率

8.2 迭代节奏

  • 第1个月：每周更新（快速修bug+小改进）
  • 第2-3个月：双周更新（功能迭代）
  • 之后：月更新（稳定节奏）
  • 每次更新App Store "What's New"认真写（用户会看）

8.3 放弃标准

如果满足以下条件，果断放弃转做下一个：

  • 上线1个月 DAU < 50 且增长趋势平坦
  • 上线2个月 付费用户 < 10
  • 用户反馈一致表示"不需要"（不是"还需要改进"）

使用方法

每次开始新App项目时：

复制此文档到项目文件夹

在立项阶段回答"5个问题"

开发过程中逐项检查checklist

上线前走一遍"发布前Checklist"

发布后按"发布后标准"执行

记住：宁可功能少但每个都精致，也不要功能多但粗糙。

用户记住的是体验，不是feature list。
